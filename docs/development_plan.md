# 통합 AI 서비스 개발 계획

## 목표

현재 하나의 FastAPI 앱에서 제공하는 매출 분석, 고객 대응방안 추천·검증, 영수증 OCR 기능을
Spring Boot와 안정적으로 연동하고 실제 운영 데이터로 학습·평가할 수 있는 수준으로 발전시킨다.

```text
Spring Boot
  → FastAPI 매출 분석
  → 분석 결과 저장 및 선택적 추천
  → 담당자 승인·수정·거절
  → 캠페인 실행 결과 수집
  → 효과 평가 및 정책 학습

Spring Boot
  → FastAPI 영수증 OCR·비용 분석
  → 구조화 결과 검증 및 저장
```

## 현재 완료된 범위

- 매출 CSV 업로드·전처리·진단·외부 요인 분석
- 사용자·매장별 분석 결과 SQLite 저장 및 소유권 검사
- LangGraph 기반 추천·효과 추정·RAG·OPE·승인 흐름
- Bandit 저장·온라인 업데이트·후보 재학습
- 캠페인 로그 계약 검증과 SCM 실측 효과 계산
- OCR·이상 지출·예산·원가율·HTML 리포트 API
- 매출 분석·추천·OCR의 단일 FastAPI 및 8000 포트 통합
- 일반 테스트와 OCR 비통합 테스트

## 1단계: API 계약과 Swagger 정리

### 작업

- Router 태그를 `매출 분석`, `전략 추천`, `캠페인 학습`, `OCR`로 구분한다.
- Spring Boot가 사용하는 요청·응답 DTO를 OpenAPI 기준으로 확정한다.
- camelCase와 snake_case 호환 기간 및 제거 시점을 결정한다.
- 모든 오류를 공통 JSON 형식으로 반환한다.
- deprecated `POST /api/v1/reports`, `POST /api/v1/agent-runs` 제거 조건을 정한다.
- 업로드 파일 크기, MIME type, 처리 timeout 기준을 추가한다.

### 완료 조건

- Spring Boot DTO와 OpenAPI schema 비교 테스트가 통과한다.
- Swagger에서 사용자 흐름 순서와 필수 헤더를 바로 확인할 수 있다.
- 신규 연동 코드가 deprecated API를 호출하지 않는다.

## 2단계: 인증 및 서비스 간 신뢰 구축

### 작업

- 사용자가 직접 입력한 `X-User-Id`만 신뢰하지 않도록 인증 계약을 변경한다.
- Spring Boot가 전달한 JWT를 검증하거나 내부 서비스 토큰을 적용한다.
- 인증 정보에서 사용자 ID와 역할을 추출한다.
- 분석·추천·캠페인·OCR API별 접근 권한을 정의한다.
- 인증·인가 실패 응답을 공통 오류 계약에 포함한다.

### 완료 조건

- 다른 사용자 ID를 임의로 전달해도 분석·추천 이력에 접근할 수 없다.
- Spring Boot를 거치지 않은 내부 API 요청을 정책에 따라 차단할 수 있다.
- 정상·만료·변조 토큰에 대한 통합 테스트가 통과한다.

## 3단계: Spring Boot 종단간 연동

### 작업

- 매출 분석 → MySQL 저장 → 선택적 추천 → 승인 흐름을 실제 RestClient로 검증한다.
- OCR 업로드 → 구조화 결과 → 백엔드 저장 흐름을 검증한다.
- FastAPI timeout, 연결 실패, 4xx, 5xx 응답 매핑을 확인한다.
- `analysis_id`, `thread_id`, `decision_id`를 양쪽 로그에 공통으로 남긴다.
- 백엔드와 FastAPI의 base URL을 `FASTAPI_BASE_URL` 하나로 통일한다.

### 완료 조건

- 실제 Spring Boot 테스트에서 매출 분석과 추천 승인까지 완료된다.
- OCR 결과가 백엔드 DTO로 손실 없이 역직렬화된다.
- FastAPI 장애 시 백엔드가 정의된 오류 코드와 사용자 메시지를 반환한다.

## 4단계: OCR 실제 모델 검증과 비동기 처리

### 작업

- `requirements-ocr.txt` 환경에서 PaddleOCR integration 테스트를 실행한다.
- 영수증 유형별 인식 정확도와 필드별 정답률을 측정한다.
- 첫 모델 로딩 시간, 요청 처리 시간, 메모리 사용량을 기록한다.
- OCR CPU 작업을 thread pool 또는 작업 큐로 분리한다.
- 동시 요청 제한, 처리 timeout, 대용량·손상 이미지 검증을 추가한다.
- OCR 실패 원인을 파일 오류, 모델 오류, 필드 검증 오류로 구분한다.

### 완료 조건

- 대표 영수증 데이터셋에 대한 정확도 기준을 문서화하고 충족한다.
- OCR 실행 중에도 매출 분석·추천 API 응답성이 유지된다.
- 다중 요청과 잘못된 파일 테스트가 안정적으로 통과한다.

## 5단계: 장시간 작업 관리

### 작업

- 매출 분석, 추천 리포트, OCR을 동기 처리할지 비동기 처리할지 기준을 정한다.
- 장시간 작업에 `job_id`와 `queued/running/completed/failed` 상태를 제공한다.
- 작업 취소, timeout, 재시도, 중복 요청 방지 정책을 구현한다.
- 초기에는 thread pool을 사용하고 운영 부하에 따라 Redis 기반 작업 큐를 검토한다.

### 완료 조건

- HTTP 연결이 종료돼도 승인된 작업 상태와 결과를 다시 조회할 수 있다.
- 동일 요청의 중복 실행 여부를 식별할 수 있다.
- 실패 원인과 재시도 횟수가 저장된다.

## 6단계: 캠페인 결과 자동 연계

### 작업

- 승인된 추천의 실행 여부·시점·대상 매장을 백엔드 데이터와 연결한다.
- 처치 전후 관측 기간과 매출 기준을 고정한다.
- 신규 분기 데이터 수집 후 실제 `revenue_after`와 reward를 자동 확정한다.
- 로그 품질 검사를 통과한 표본만 SCM·OPE·Bandit에 전달한다.
- 사용자·매장별 추천 이력과 실제 효과 조회 API를 제공한다.

### 완료 조건

- 추천부터 실행 결과까지 `analysis_id`, `thread_id`, `decision_id`로 추적된다.
- 수동 CSV 수정 없이 실제 매출과 추천 결과가 연결된다.
- 효과 평가 결과에 표본 수, 신뢰구간, 판정 근거가 포함된다.

## 7단계: 모델 운영과 승격

### 작업

- 합성 로그와 실제 로그를 분리해 학습·평가한다.
- 후보 Bandit을 기존 active 모델과 같은 검증 데이터에서 비교한다.
- OPE, ESS, 신뢰구간과 사용자·매장별 성과를 함께 확인한다.
- 후보 모델을 shadow mode로 운영한 후 승인된 모델만 승격한다.
- 데이터 범위, feature schema, 모델 버전, 평가 결과를 저장한다.
- 성능 저하 시 이전 모델로 복구하는 절차를 만든다.

### 완료 조건

- 어떤 데이터와 코드로 모델이 생성됐는지 재현할 수 있다.
- 자동 승격 없이 검증·승인 절차를 통과한 모델만 active가 된다.
- 롤백 후 추천 정책이 이전 버전과 동일하게 복구된다.

## 8단계: 관측성 및 운영 저장소

### 작업

- 모든 요청에 `request_id` 또는 `trace_id`를 부여한다.
- 구조화 JSON 로그에 처리 단계, 시간, 모델·정책 버전을 기록한다.
- `/health`와 모델·데이터 준비 상태를 확인하는 `/ready`를 분리한다.
- OpenAI·외부 데이터 API·OCR 호출의 시간과 실패율을 측정한다.
- SQLite 업무 데이터의 MySQL 이관 범위를 결정한다.
- 여러 FastAPI 인스턴스에서 공유해야 하는 상태를 외부 저장소로 이전한다.

### 완료 조건

- 한 요청의 Spring Boot와 FastAPI 로그를 같은 trace ID로 조회할 수 있다.
- 모델·데이터가 준비되지 않은 인스턴스는 트래픽을 받지 않는다.
- 장애 원인과 영향을 받은 분석·추천 실행을 식별할 수 있다.

## 9단계: CI/CD 및 배포

### 작업

- PR마다 일반 테스트, 정적 검사, OpenAPI 계약 검사를 실행한다.
- 데이터·모델·PaddleOCR이 필요한 integration 테스트를 별도 job으로 분리한다.
- 기본 및 OCR 의존성을 포함한 통합 Docker 이미지를 구성한다.
- 환경변수 누락과 잘못된 base URL을 시작 단계에서 검증한다.
- 배포 전 smoke test로 `/health`, `/ready`, 핵심 API를 확인한다.

### 완료 조건

- 신규 PR이 API 계약이나 테스트를 깨면 병합이 차단된다.
- 동일 이미지로 개발·검증·운영 환경을 실행할 수 있다.
- 배포 실패 시 이전 버전으로 복구할 수 있다.

## 권장 진행 순서

1. API 계약과 Swagger 정리
2. 인증 및 Spring Boot E2E 연동
3. OCR 실제 모델 integration 테스트
4. 장시간 작업 비동기화
5. 캠페인 결과 자동 연계
6. 모델 승격·관측성·CI/CD 구축

새 모델 추가보다 현재 기능의 실제 연동, 신뢰 경계, 데이터 추적성을 먼저 완성한다.
