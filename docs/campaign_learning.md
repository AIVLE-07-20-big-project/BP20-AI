# 캠페인 학습 및 효과 검증

## 목적과 흐름

승인된 추천과 실제 실행 결과를 연결해 Synthetic Control, OPE, Bandit이 같은 로그 계약을
사용하도록 한다.

```text
추천·승인 → 실행 결과 기록 → 로그 검증 → SCM·OPE 평가 → Bandit 업데이트 → 후보 모델 검증
```

## 로그 스키마

한 행은 특정 상권·업종에서 정책이 방안을 선택하고 실행한 결과 한 건이다.

| 컬럼 | 정의 |
|---|---|
| `decision_id` | 결정 고유 ID |
| `user_id`, `store_id` | 사용자·매장 소유권 식별자 |
| `trdar_cd`, `svc_induty_cd` | 상권·업종 코드 |
| `yyqu_cd`, `treatment_yyqu_cd` | 관측 분기와 실행 분기 |
| `action_id` | 실행한 대응방안 |
| `context_1`~`context_6` | 결정 시점 Bandit 컨텍스트 |
| `propensity` | 당시 정책의 선택 확률 |
| `policy_version` | 정책 버전 |
| `executed` | 실제 실행 여부 |
| `revenue_before`, `revenue_after` | 실행 전후 매출 |
| `reward` | `(after - before) / before` |
| `데이터_출처` | `synthetic` 또는 `real` |

`propensity`는 사후 복원할 수 없으므로 추천 시점 값을 저장한다. 미실행 건은 사후 매출과
reward를 비워 두며 실행 이전 컨텍스트에 미래 데이터를 섞지 않는다.

## 검증 규칙

- 필수 컬럼·타입·`decision_id` 중복 확인
- 알려진 `action_id`와 `0 < propensity <= 1` 확인
- 실행 분기가 관측 분기보다 이후인지 확인
- 실행 여부와 보상 결측 상태 확인
- 저장 reward와 매출 변화율 재계산 결과 비교
- 요청 사용자와 승인 실행의 소유권 확인

제외된 행은 SCM·OPE·Bandit 학습에 사용하지 않는다.

- 로그 수집 및 품질 API: 완료
- SCM 실측 효과와 부트스트랩 신뢰구간: 완료
- IPS·DM·DR 및 ESS 기반 정책 평가: 완료
- 등급별 Bandit 저장·온라인 업데이트·후보 재학습: 완료

- 후보 모델 자동 승격: 미구현, 수동 승인 유지

## 합성 로그 원칙

- 실제 상권·업종·분기 셀을 사용한다.
- 모든 행에 `데이터_출처=synthetic`을 기록한다.
- propensity를 균등 상수로 만들지 않는다.
- 근거 없는 방안 효과를 임의로 만들지 않는다.
- 실제 로그가 충분해지면 합성 행을 분리하거나 폐기한다.
- 합성 표본을 늘려 모델 확신을 인위적으로 높이지 않는다.

## 다음 개발 단계

1. 신규 분기 데이터 수집·전처리·품질 검사를 자동화한다.
2. 승인된 추천의 실행 시점과 사후 매출을 자동 연결한다.
3. 결과 확정 후 SCM과 OPE를 자동 실행해 저장한다.
4. 사용자·매장별 추천 이력과 실제 성과를 함께 조회한다.
5. 실제 표본 기준을 충족하면 후보 Bandit을 shadow mode로 평가한다.
6. 안전성이 검증된 후보만 수동 승인으로 승격한다.

실제·합성 로그 성능을 섞어 보고하지 않고 표본 수, SCM 적합도, OPE 신뢰구간과 ESS를 함께
판단한다. 학습 데이터 범위와 모델 버전은 재현 가능하게 기록한다.

자동 연계와 모델 승격의 단계별 완료 조건은 `development_plan.md`의 6~8단계를 따른다.
